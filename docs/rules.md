# RAKSHAK — Build Rules

## Always do
- Keep scoring, fusion, and ATT&CK matching **deterministic** (plain Python/graph logic). No LLM
  calls anywhere in that path.
- Every fused score must log **which sources contributed and by how much**. No silent scores.
- Every score is presented as belief/plausibility/uncertainty (or an equivalent triple) —
  never collapse evidence into one bare float in the UI.
- OT nodes: active disruption actions must be hard-blocked in code (not just in the UI), with a
  mandatory human-escalation flag on any OT-adjacent action.
- Keep IT graph and OT graph as separate data structures; only allow correlation through explicit
  `IT_OT_BRIDGE` node types.
- Use realistic, internally-consistent synthetic data — the scripted incident should read as a
  coherent APT campaign (Initial Access → ... → Impact), not random noise.
- Keep components small and single-purpose; Living Graph rendering logic stays isolated from data
  fetching logic.
- Write the DS combination and resilience score formulas as pure, unit-testable functions.

## Never do
- Never train or claim to train a real ML model for the hackathon build — stand-ins must be
  clearly deterministic/statistical, not misrepresented as trained models in code comments.
- Never let the LLM output affect scoring, matching, or the response gate's decision — LLM output
  is display-only narration or a reviewable suggestion.
- Never hardcode the "answer" behind the AI query bar — it must be generated by prompting the
  LLM with the actual structured evidence retrieved for that entity/cluster.
- Never mix IT and OT nodes in the same graph object.
- Never store API keys in source files — use environment variables only.
- Never add a UI library or component kit beyond Tailwind + hand-built primitives — the
  glassmorphism look must be custom CSS, not a templated kit, to avoid looking generic.
- Never let a single phase's prompt touch more than the files/folders scoped to that phase.
- Never introduce a database (Postgres/Mongo) for the hackathon MVP — NetworkX + JSON files are
  sufficient and reduce failure surface.
- Never block progress waiting on Neo4j — it's explicitly a stretch goal only.

## Code quality bar
- TypeScript strict mode on frontend; type hints on all backend function signatures.
- Every backend module under `fusion/`, `scoring/`, `resilience/` needs at least one test case with
  a known input/output pair.
- No inline styles in React components beyond dynamic values (e.g. animation timing) — static
  styling goes in Tailwind classes or the CSS module.
